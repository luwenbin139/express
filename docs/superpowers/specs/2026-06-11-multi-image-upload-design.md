# 多图片上传与前端重建设计

## 背景

当前项目是一个 Express 服务，静态托管 `build/` 前端产物。图片生成接口已存在于 `index.js`，但只支持单张 base64 data URL 图片：`image: "data:image/..."`。仓库中缺少可维护的前端源码，只有已有的 `build/` 构建产物。

## 目标

1. 后端图片接口支持一次上传多张图片。
2. 保持旧客户端兼容：继续支持单张 `image` 字段。
3. 新增 multipart/form-data 文件上传能力，接口字段名为 `images`。
4. 新增可维护前端源码，并构建输出到根目录 `build/`。
5. 前端支持最多选择 4 张图片，每张最大 10MB。

## 不做的范围

- 不做图片持久化存储，上传图片只在请求处理中转为 data URL 后传给 OpenAI。
- 不改数据库结构。
- 不保留或逆向修改旧 `build/` 产物里的页面逻辑；新前端构建会替换现有 `build/`。

## 后端设计

### 接口

继续使用现有接口：

- `POST /api/generate-image`
- `POST /api/generate-image-stream`

两者都支持以下请求格式。

### JSON 格式

兼容旧格式：

```json
{
  "prompt": "...",
  "size": "1024x1024",
  "mode": "edit",
  "image": "data:image/png;base64,..."
}
```

新增多图格式：

```json
{
  "prompt": "...",
  "size": "1024x1024",
  "mode": "edit",
  "images": [
    "data:image/png;base64,...",
    "data:image/jpeg;base64,..."
  ]
}
```

如果同时传 `image` 和 `images`，后端合并并去除空值，再统一限制最多 4 张。

### Multipart 格式

请求类型：`multipart/form-data`

字段：

- `prompt`: 文本提示词
- `size`: 图片尺寸
- `mode`: `generate` 或 `edit`
- `images`: 多文件字段，最多 4 个文件

限制：

- 最多 4 张图片
- 每张最大 10MB
- 仅允许 `image/png`、`image/jpeg`、`image/webp`

### 校验与错误处理

- `mode=edit` 时至少需要 1 张图片。
- 图片数量超过 4 张返回 400。
- 图片格式不支持返回 400。
- 单张图片超过 10MB 返回 400。
- OpenAI 调用失败时保持现有错误返回/事件格式。

### OpenAI Payload

现有 payload 构建逻辑从单个 `imageDataUrl` 改为 `imageDataUrls` 数组。每张图片添加一个：

```js
{
  type: 'input_image',
  image_url: dataUrl
}
```

## 前端设计

### 源码位置

新增 `frontend/` 目录，作为可维护前端源码目录。构建输出目录配置为项目根目录 `build/`。

### 页面功能

页面重新布局为“配置区 + 状态/结果区”：

- 桌面端：左侧为配置区，右侧为图片预览、生成状态和结果区。
- 移动端：上下布局，先配置，再状态与结果。

页面提供：

- 提示词输入框
- 模式选择：生成 / 编辑
- 图片尺寸选择
- 多图片选择控件
- 已选图片预览与移除
- 提交按钮
- 结果图片展示
- 错误提示
- 生成计时器
- 取消生成按钮
- 长耗时提示：生成通常需要 5-10 分钟，当前页面关闭后任务会中断

### 上传行为

前端默认使用 `multipart/form-data` 调用流式接口 `POST /api/generate-image-stream`：

- 文件字段名：`images`
- 最多 4 张
- 每张最大 10MB
- 仅允许 png/jpeg/webp

前端在提交前做一次基础校验，后端仍做最终校验。

流式等待体验：

- 提交后立即展示“已提交请求”。
- 收到 SSE `status` 后展示当前阶段。
- 收到 SSE `heartbeat` 后更新“最近连接正常时间”。
- 收到 SSE `partial_image` 后展示局部图。
- 收到 SSE `final_image` 或 `done` 后展示最终图。
- 收到 SSE `error` 后停止等待并展示可理解错误。
- 用户点击取消时，前端中断当前请求。

本期不做后台任务恢复。用户关闭页面或刷新页面后，当前生成任务会中断或丢失。代码结构预留后续扩展任务 ID 的空间，但不引入数据库或持久化。

### 构建与部署

新增前端构建脚本，构建产物直接输出到根目录 `build/`。Express 继续通过现有静态目录逻辑托管 `build/`。

## 测试与验证

至少验证：

1. 旧 JSON 单图 `image` 请求仍可用。
2. 新 JSON 多图 `images` 请求可用。
3. multipart 多图请求可用。
4. 超过 4 张图片返回 400。
5. 非图片文件返回 400。
6. 流式接口生成期间会周期性发送 `heartbeat`，避免 5-10 分钟长耗时生成时连接空闲超时。
7. 客户端断开时，服务端会销毁上游请求。
8. 前端默认使用流式接口，并展示清晰等待状态、计时器、最近心跳时间和取消按钮。
9. 前端可构建到 `build/`。
10. Express 可继续托管构建后的前端。

## 风险与处理

- base64 JSON 多图会增大请求体：保留现有 JSON 支持，但新前端优先使用 multipart。
- 新构建会替换旧 `build/` 页面：这是本次方案的明确范围。
- 缺少现有前端源码：通过新增 `frontend/` 解决后续维护问题。
- 图片生成可能需要 5-10 分钟：通过 SSE 状态事件和 `heartbeat` 保持连接活跃，同时前端明确展示等待状态。
- 本期没有数据库：不做后台任务恢复，避免引入不可靠的内存任务状态；后续有持久化后再扩展任务 ID 查询。
